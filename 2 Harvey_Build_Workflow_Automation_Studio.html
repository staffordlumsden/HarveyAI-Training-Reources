Start with a repeatable task, not the automation tool.

Build is most useful when the task is structured, repeatable, reviewable, and bounded by clear inputs, outputs, limits, and verification points.

Purpose

This studio helps educators prepare a plain-English workflow-agent description before testing it in Harvey Build.

Working principle

Use Build for workflows that support drafting, organising, extracting, comparing, or checking material for human review.

Boundary

Do not use Build to replace legal judgement, assessment judgement, source checking, confidentiality decisions, or policy compliance.

Selection rule

Choose a workflow only if it can be tested with low-risk materials, checked against sources or criteria, and revised before being shared.

Select one Build use case.

Select the workflow that gives you a useful automation test while keeping the task low-risk, reviewable, and connected to teaching, assessment, or administrative support.

Good candidates

Suitable tasks include seminar-question generation, lecture-note updating, rubric drafting, feedback-theme drafting, prompt governance, workflow mapping, and moderation checklists.

What to avoid

Avoid workflows that require final legal advice, confidential judgement, unredacted student data, final grades, or decisions that must remain entirely human-led.

Task-selection prompt

Ask: “Have I done this task often enough, and is the output easy enough to review, that a workflow agent would actually save time?”

Describe the workflow in plain English.

Build works best when the workflow description names the user, task, inputs, jurisdiction, output format, limits, and human review points.

Draft Build description

Test the agent with low-risk material and verify the output.

Before sharing the workflow, test it with public, dummy, or otherwise low-risk materials. Check whether the output follows the workflow, respects the limits, and identifies matters requiring human review.

Test material

Use non-sensitive, already-public, dummy, or approved materials. A public judgment is safer than student work for the first test.

Verification focus

Check citations, legal propositions, quotations, classifications, assumptions, and any suggested teaching use.

Revision rule

If the output overreaches, narrow the task, strengthen limits, add source requirements, or require a clearer review section.

Human review required

A workflow agent is ready only when a user can explain what it does, what it needs, what it must not do, and how the output will be verified before use.

Share carefully and review periodically.

After testing, decide whether the workflow remains personal, is shared with a teaching team, or is proposed as a faculty resource. Do not share broadly until it has been tested and risk-checked.

Collegial review

Ask a colleague to review the agent instructions or test the output. Look for overbroad wording, missing limits, unclear input requirements, and weak verification steps.

Maintenance

Revisit the workflow when law, policy, assessment design, unit materials, or Harvey functionality changes.

            Seminar-question example
            
          

          
Create a workflow agent to support academic staff preparing seminar questions for an Australian law unit.

The user will provide the unit name, student cohort, jurisdiction, topic, and either a case summary, judgment extract, or short problem scenario.

The workflow should:
1. Confirm the teaching aim.
2. Identify the legal topic and assumed student level.
3. Generate five discussion questions moving from recall to application and evaluation.
4. Include one likely student misconception.
5. Include one source-checking reminder.
6. Avoid producing model answers.
7. End with a short educator review checklist.

The workflow must not invent authorities, provide legal advice, or suggest that the output is ready for use without educator review.

        


        

          

            Assessment-feedback example
            
          

          
Create a workflow agent to support educators drafting feedback themes from anonymised student submissions.

The user will provide the assessment instructions, rubric or criteria, and anonymised extracts or summaries of student work.

The workflow should:
1. Confirm that no names, student IDs, or other PII are included.
2. Identify the relevant assessment criteria.
3. Map observed strengths and weaknesses to the supplied criteria.
4. Draft cohort-level feedback themes.
5. Suggest feed-forward advice for future work.
6. Avoid assigning individual marks.
7. Flag any judgement requiring academic moderation.

The output should be a table with: criterion, observed pattern, evidence from supplied material, draft feedback theme, and human review note.
